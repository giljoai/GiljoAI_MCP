# Security Policy

## Supported Versions

Active support is on the 1.1.x line; older majors are not maintained.

| Version | Supported          |
|---------|--------------------|
| 1.1.x   | Yes (current)      |
| < 1.1   | No                 |

## Reporting a Vulnerability

If you discover a security vulnerability, please report it responsibly:

1. **Do NOT** open a public GitHub issue for security vulnerabilities
2. Email security concerns to **abuse@giljo.ai**
3. Include a description of the vulnerability, steps to reproduce, and potential impact

## Response Timeline

- **Acknowledgment**: Within 48 hours of report
- **Assessment**: Within 7 days
- **Fix/Mitigation**: Depends on severity (critical: 48h, high: 7 days, medium: 30 days)

## Security Architecture

### Application

- **Authentication**: Always enabled, JWT-based with bcrypt password hashing
- **No default credentials**: First user created via `/welcome` setup wizard
- **Multi-tenant isolation**: Per-user tenant keys with database-level filtering on every query
- **API authentication**: `X-API-Key` header for MCP tool access
- **Password reset**: Recovery PIN system with rate limiting
- **WebSocket security**: JWT token required for all connections
- **Network**: Localhost binds 127.0.0.1 (HTTP). LAN/WAN binds 0.0.0.0 with mandatory HTTPS (mkcert)

### Secrets and Source-Code Defense

A four-layer defense prevents secrets from reaching the public repo:

1. **Pre-commit (private repo)** — `gitleaks` runs on staged content via `.pre-commit-config.yaml`. Blocks at staging time.
2. **Push CI (private repo)** — Backend CI / Secret scan job re-runs `gitleaks` on every push and PR. Catches anything that bypassed pre-commit (e.g., `--no-verify`).
3. **Boundary gate (export scripts)** — Both `scripts/export_ce.sh` and `scripts/export_ce_dev.sh` run `gitleaks detect --no-git` against the SaaS-stripped tree **after strip and before push to public**. Last-chance scan.
4. **Public CI (public repo)** — Defensive `gitleaks` working-tree scan on the public repo. Should never trigger if layers 1-3 hold; if it does, fix upstream.

If any layer triggers in production, treat as a layer-1-3 gap and harden upstream.

### Dependency Security

- Python dependencies are pinned in `requirements.txt`; dev dependencies in `pyproject.toml` `[project.optional-dependencies] dev`
- Frontend dependencies managed via `package-lock.json`
- Dependabot configured for `pip`, `npm`, and `github_actions` ecosystems on the private repo (weekly)
- Major version bumps are triaged individually; deferred majors stay on Dependabot's queue (no silent ignores)

### Supply-Chain

- All commits to public master flow through the export pipeline (no direct pushes); the squash-merge model preserves a clean public history
- AI signatures (`Co-Authored-By: claude/anthropic/openai/copilot/cursor`, `Generated by`, `AI-generated`) are blocked by both pre-commit hook and CI policy check
- License (`GiljoAI Community License v1.1`) is enforced as a header on Python and Vue source files

## CI Architecture

- **Private repo** is the primary CI gate (9 required checks via branch-protection ruleset)
- **Public repo** runs a slimmed 6-check smoke set (defense-in-depth)
- All exports require private CI green before propagation to public
