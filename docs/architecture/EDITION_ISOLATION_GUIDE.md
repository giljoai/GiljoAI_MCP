## A. Edition Model

GiljoAI MCP is a **single codebase** in a **single private repository**. Three deployment modes are controlled by the `GILJO_MODE` environment variable.

| Mode | `GILJO_MODE` | Visibility | Users | Install Method |
|------|-------------|-----------|-------|---------------|
| Community (CE) | unset (default) | Public repo (exported) | Single user | `git clone` + `python install.py` |
| Demo | `demo` | Private, hosted by GiljoAI | Multi-user trial (7-day) | Server-side provisioning |
| SaaS | `saas` | Private, hosted by GiljoAI | Multi-user, multi-org, subscription | Docker/K8s |

**Key relationships:**
- **Demo is SaaS Lite.** Demo uses real multi-tenant infrastructure (registration, provisioning, org setup) with a 7-day TTL. Everything built for Demo is reused by SaaS.
- **Enterprise** is a deployment mode of SaaS (self-hosted by corporate IT), not a separate edition.
- The Community Edition is licensed under the **GiljoAI Community License v1.1** (single-user free, multi-user requires commercial license). This is NOT an OSI-approved license. Never reference MIT, open source, or open core in code, documentation, or UI.

## B. Repository Architecture

**Two repos, one codebase:**

| Repo | Visibility | Purpose |
|------|-----------|---------|
| `giljoai/GiljoAI_MCP_Private` | **Private** | Source of truth. ALL code: CE + Demo + SaaS |
| `giljoai/GiljoAI_MCP` | **Public** | CE downstream. Generated by `scripts/export_ce.sh` |

**All development happens in the private repo.** The public repo is a filtered export -- never commit to it directly (except community PRs, which get cherry-picked back).

**Deployment flow:**

```
Private repo --> scripts/export_ce.sh --> Public repo (CE, normal push)
Private repo --> deploy with GILJO_MODE=demo --> Demo server
Private repo --> deploy with GILJO_MODE=saas --> SaaS server
Public repo <-- Community PR --> cherry-pick back to Private
```

**Export script** (`scripts/export_ce.sh`): clones public repo, replaces working tree with filtered private content (strips `saas/`, `demo/`, `handovers/`, `scripts/`, internal docs, AI tool configs), normal push. See `.export-exclude` for full exclusion list.

## D. The Four Rules

These are mandatory rules, not suggestions. Every agent must follow them without exception.

### Rule 1 -- Import Direction

SaaS/Demo code may import from CE code (they extend CE). CE code NEVER imports from any `saas/`, `demo/`, `saas_endpoints/`, or `saas_middleware/` directory. If you find yourself needing CE to call SaaS code, use the conditional registration pattern (see Section F), never a direct import.

**Verification command:**

```bash
# Run from project root. Expected result: no output.
grep -r "from.*saas\|from.*demo" src/giljo_mcp/ api/ frontend/src/ \
  --include="*.py" --include="*.js" --include="*.vue" \
  | grep -v "saas/" | grep -v "demo/" | grep -v "__pycache__"
```

If this command produces any output, there is a dependency leak. Fix it before completing the handover.

### Rule 2 -- The Deletion Test

Before marking any SaaS or Demo work as complete, confirm: "If all `saas/`, `saas_endpoints/`, `saas_middleware/`, `frontend/src/saas/`, and `demo/` directories were deleted, would CE still start, serve requests, and pass all tests in `tests/` (excluding `tests/saas/`)?" If no, there is a dependency leak. Fix it before completing the handover.

### Rule 3 -- Placement Decision Framework

Use this decision tree for every new feature:

```
Does this feature require infrastructure a solo developer wouldn't have?
(Stripe account, SMTP relay, LDAP server, OAuth provider)
    --> YES --> saas/ directory

Is this Demo trial management? (registration, tenant provisioning,
7-day TTL reaper, trial expiry emails)
    --> YES --> saas/ directory (Demo reuses SaaS provisioning code)

Does this feature only make sense with multiple users or organizations?
(team management, viewer roles, cross-user product transfer, org analytics)
    --> YES --> saas/ directory

Does this feature improve the core orchestration for ANY user, including solo?
(better mission planning, new MCP tools, UI improvements, performance)
    --> YES --> CE directory

Does this code already exist in CE and is needed for CE to function?
(TenantManager, Organization model, multi-user auth infrastructure)
    --> YES --> stays in CE directory, do NOT move to saas/

Is this Demo-only configuration? (config overrides, demo deployment docs)
    --> YES --> demo/ directory
```

**Important:** Demo trial infrastructure (registration, provisioning, reaper) lives in `saas/` not `demo/` because SaaS reuses it directly. The `demo/` directory is only for Demo-specific configuration that SaaS would not use.

### Rule 4 -- Handover Tagging

Every handover document MUST include in its metadata:

```
**Edition Scope:** CE | Demo | SaaS | Both
```

This tells the receiving agent which directories they should be working in.

## E. Existing SaaS-Adjacent Code (Do NOT Move)

The following code is intentionally in CE directories and must NOT be relocated to `saas/`. These components serve dual purposes: they enable CE to function in single-user/single-tenant mode, and they provide the foundation that Demo and SaaS extend.

| Component | Location | Why It Stays in CE |
|-----------|----------|-------------------|
| TenantManager | `src/giljo_mcp/tenant.py` | CE uses it in hidden single-tenant mode. Demo/SaaS activate multi-tenant. |
| Organization model | `src/giljo_mcp/models/` | CE creates a single implicit org at install. Demo/SaaS activate multi-org. |
| Multi-user auth | `src/giljo_mcp/auth/` | CE uses single-user login. SaaS extends it (OAuth, MFA). |
| User management endpoints | `api/endpoints/users.py` | CE manages the single admin user. SaaS adds team features. |
| API key system | Across auth + endpoints | CE uses API keys for MCP tool auth. SaaS adds per-org key management. |
| WebSocket broker (PostgresNotify) | `api/broker/` | CE uses it for single-instance real-time. SaaS uses it for multi-instance. |
| Frontend tenant awareness | `frontend/src/` (X-Tenant-Key headers) | CE sends implicit key. SaaS adds org switcher. |
| Setup wizard | `frontend/src/components/setup/` | CE uses for first-run setup. Demo/SaaS reuses for MCP tool connection after registration. |

The principle: if CE needs this code to boot and function in single-user mode, it stays in CE even though Demo and SaaS will also use it.

## F. Conditional Loading Patterns

When SaaS features need to integrate with the CE application without creating import dependencies, use these conditional loading patterns. CE code checks whether directories exist at runtime and loads modules only if present. If the directory is missing (CE-only deployment), the code silently skips.

### Backend (FastAPI app.py) -- GILJO_MODE detection

```python
import os

GILJO_MODE = os.environ.get("GILJO_MODE", "ce")  # "ce", "demo", "saas"
```

### Backend -- Conditional endpoint registration

```python
# CE endpoints -- always loaded
from api.endpoints import auth, projects, agents, orchestrator, messages
app.include_router(auth.router)
app.include_router(projects.router)
# ... etc

# SaaS/Demo endpoints -- loaded only if saas_endpoints directory has content
_saas_endpoints_dir = os.path.join(os.path.dirname(__file__), "saas_endpoints")
if os.path.isdir(_saas_endpoints_dir) and GILJO_MODE in ("demo", "saas"):
    try:
        from api.saas_endpoints import registration, org_setup, trial
        app.include_router(registration.router, prefix="/api/saas", tags=["SaaS: Registration"])
        app.include_router(org_setup.router, prefix="/api/saas", tags=["SaaS: Org Setup"])
        app.include_router(trial.router, prefix="/api/saas", tags=["SaaS: Trial"])
    except ImportError:
        pass  # SaaS modules not available -- CE mode

    # SaaS-only (not demo)
    if GILJO_MODE == "saas":
        try:
            from api.saas_endpoints import billing, analytics
            app.include_router(billing.router, prefix="/api/saas", tags=["SaaS: Billing"])
            app.include_router(analytics.router, prefix="/api/saas", tags=["SaaS: Analytics"])
        except ImportError:
            pass
```

### Backend -- Conditional middleware registration

```python
if GILJO_MODE in ("demo", "saas"):
    try:
        from api.saas_middleware.trial_enforcement import TrialEnforcementMiddleware
        app.add_middleware(TrialEnforcementMiddleware)
    except ImportError:
        pass
```

### Frontend (Vue Router) -- Conditional route loading

```javascript
// router/index.js

// CE routes -- always loaded
import DashboardView from '@/views/DashboardView.vue'
// ... other CE imports

const routes = [
  { path: '/dashboard', component: DashboardView },
  // ... other CE routes
]

// SaaS routes -- loaded only if saas/ directory exists
try {
  const saasRoutes = require('@/saas/routes.js').default
  routes.push(...saasRoutes)
} catch (e) {
  // SaaS routes not available -- CE mode
}
```

### Alembic Migrations -- SaaS-only migration branch

SaaS-only database tables get their own migration directory. The CE migration chain (`migrations/versions/`) remains untouched and self-contained.

```python
# In migrations/env.py
import os

# Always run CE migrations
version_locations = "migrations/versions"

# Add SaaS migrations if directory exists and has content
saas_migrations = os.path.join(os.path.dirname(__file__), "saas_versions")
if os.path.isdir(saas_migrations):
    version_locations += f" {saas_migrations}"

context.configure(
    version_locations=version_locations.split(),
    # ... other config
)
```

### Migration placement rules

- If a SaaS/Demo feature needs a **new column on an existing CE table**, that migration goes in the CE chain (`migrations/versions/`) because both editions benefit from schema compatibility.
- If a SaaS/Demo feature needs an **entirely new table**, it goes in `migrations/saas_versions/`.

## G. Demo Mode -- Architecture & User Flow

Demo mode (`GILJO_MODE=demo`) is the bridge between CE and SaaS. It uses real SaaS infrastructure with a 7-day trial limit.

### User Journey

1. **Registration:** User submits email on giljo.ai. Backend provisions a new tenant with a random password.
2. **Welcome email:** User receives login URL + credentials via transactional email.
3. **First login:** Org setup screen ("Welcome, set up your organization and admin account").
4. **Setup wizard:** Connect CLI tools via MCP (same wizard as CE).
5. **Normal usage:** Create product, create project, run agents -- full platform for 7 days.
6. **Expiry:** After 7 days, tenant is reaped. User receives "trial expired" email with upgrade link.

### What Demo Shares with SaaS

| Component | Demo | SaaS |
|-----------|------|------|
| Email registration | Yes | Yes |
| Tenant provisioning | Yes (auto, random password) | Yes (user-chosen password) |
| Org setup first-login | Yes | Yes |
| Password reset | Yes | Yes |
| 2FA (TOTP) | Groundwork (DB + backend) | Full (UI + enforcement) |
| Trial TTL + reaper | Yes (7-day hard limit) | Yes (becomes subscription expiry) |
| SSL termination | Cloudflare Tunnel (when public) | Load balancer / Cloudflare |
| Setup wizard (MCP tools) | Reuses CE wizard | Reuses CE wizard |

### What Demo Does NOT Have

- Stripe / billing (free trial, no payment)
- OAuth / SSO / LDAP
- Team management (single admin per trial tenant)
- Usage analytics / metering

### Hosting

- **Initial (test bed):** Self-hosted on office machine, direct IP + port forward, plain HTTP
- **Public launch:** Cloudflare Tunnel to office machine, `demo.giljo.ai`, Cloudflare terminates SSL
- **Production SaaS:** Datacenter deployment (future)

## H. CI/CD Implications

- The public repo CI runs CE tests only (`tests/` excluding `tests/saas/`).
- The private repo CI (when enabled) runs ALL tests including `tests/saas/`.
- If a test in `tests/saas/` fails, it does NOT block CE releases.
- SaaS tests may import from `saas/` directories. CE tests must NOT.

### Test isolation verification

```bash
# Verify no CE code imports from saas or demo directories
grep -r "from.*saas\|from.*demo" src/giljo_mcp/ api/ frontend/src/ \
  --include="*.py" --include="*.js" --include="*.vue" \
  | grep -v "saas/" | grep -v "demo/" | grep -v "__pycache__"
# If this produces output, the build should FAIL.
```
