## 1. System Overview

### What is GiljoAI MCP Server?

**GiljoAI Agent Orchestration MCP Server** is a multi-tenant server that orchestrates specialized AI agents for complex software development. It enables perpetual context management and intelligent agent coordination through Model Context Protocol (MCP).

### Key Capabilities

- **Multi-tenant isolation** - Secure tenant separation via API/Bearer keys
- **Agent orchestration** - Coordinate up to 8 specialist agent types
- **Database-driven prompts** - Instructions stored in PostgreSQL, fetched via MCP
- **Perpetual context** - Manage long-running projects beyond single-session limits
- **Thin client architecture** - ~10 line prompts that fetch full context from server
- **Real-time coordination** - WebSocket updates and message queue communication

### Deployment Options

- **LAN** - Local network deployment for team collaboration
- **WAN** - Wide area network for distributed teams
- **Hosted** - Cloud deployment for maximum accessibility

### Technology Stack

- **Backend:** Python 3.11+, FastAPI, SQLAlchemy
- **Frontend:** Vue 3, Vuetify, WebSockets
- **Database:** PostgreSQL 18 (required, no SQLite support)
- **Protocol:** MCP over HTTP (not STDIO)

```
│                                             │
│  ┌──────────────┐      ┌─────────────────┐ │

│         │                       │          │

│  │     Backend & PostgreSQL Database     │ │
│  └───────────────────────────────────────┘ │
└─────────────────────────────────────────────┘
           ▲                    ▲
           │                    │
    ┌──────┴──────┐      ┌──────┴──────┐
    │   Browser   │      │  CLI Tools  │
    │  (Web UI)   │      │  (MCP+HTTP) │
    └─────────────┘      └─────────────┘
```

### 2.2 Multi-User Architecture

The server supports multiple developers accessing the same centralized server:

- Each developer has their own **tenant** with isolated data
- Developers access via **Browser** (web UI) and **CLI tools** (MCP-enabled)
- Authentication via **API keys** (Claude Code, Gemini) or **Bearer tokens** (Codex)

### 2.3 Data Hierarchy

```
Tenant
└── Product (one active)
    ├── Vision Documents (multiple, chunked for RAG)
    ├── Tech Stack & Configuration
    └── Projects (one active per product)
        ├── Description (human-written requirements)
        ├── Mission (AI-generated by orchestrator)
        └── Agent Jobs (up to 8 types, unlimited instances)
            ├── Job Instructions (stored in database)
            ├── Messages (agent communication queue)
            └── Progress Tracking
```

### 2.4 Database-Driven Prompts (CRITICAL)

**How Instructions Work:**

❌ **MISCONCEPTION:** User copies massive 3000-line prompts
✅ **REALITY:** User copies ~10 line prompt that fetches instructions from database

**The Flow:**

1. User clicks "Stage Project" button in web UI
2. System generates **thin prompt** (~10 lines)
3. User pastes prompt into CLI tool terminal
4. Agent executes prompt and calls `get_orchestrator_instructions(orchestrator_id, tenant_key)`
5. MCP tool fetches **full mission from database** (stored in `MCPAgentJob.mission` field)
6. Agent receives complete context and begins work

**Database Tables Involved:**

- `MCPAgentJob.mission` - Stores agent-specific instructions
- `Project.description` - Human-written requirements (input for orchestrator)
- `Project.mission` - AI-generated mission plan (created by orchestrator)
- `VisionDocument.content` - Product vision (chunked for context)

## 4. Initial Setup Workflows

### 4.1 MCP CLI Setup (Static Copy-Paste Prompts)

**Purpose:** Attach MCP server to CLI tool (Claude Code, Codex, Gemini)

**Steps:**

1. Navigate to **My Settings → Integrations → MCP Setup**
2. Click "Generate Connection Prompt"
3. System generates tool-specific configuration:

**For Claude Code / Gemini:**
```json
{
  "mcpServers": {
    "giljo-mcp": {
      "url": "http://{server_ip}:7272/mcp/http",
      "transport": "sse",
      "headers": {
        "X-API-Key": "{your_api_key}"
      }
    }
  }
}
```

**For Codex:**
```json
{
  "mcpServers": {
    "giljo-mcp": {
      "url": "http://{server_ip}:7272/mcp/http",
      "transport": "sse",
      "headers": {
        "Authorization": "Bearer {your_bearer_token}"
      }
    }
  }
}
```

4. **Copy prompt** and paste into CLI tool configuration
5. Restart CLI tool
6. MCP server is now attached and authenticated

### 4.2 Slash Command Setup (Static Copy-Paste Prompts)

**Purpose:** Install custom slash commands for CLI tools

**Steps:**

1. Navigate to **My Settings → Integrations → Slash Commands**
2. Click "Generate Download Prompt"
3. System generates prompt with **tokenized download link** (15-min TTL)
4. Copy prompt and paste into CLI terminal
5. Agent downloads ZIP and installs slash commands
6. Restart CLI tool
7. Slash commands appear (e.g., `/gil_stage`, `/gil_launch`)

**Alternative:** Direct download link provided for manual installation

### 4.3 Agent Template Download (Claude Code Only - Static Copy-Paste)

**Purpose:** Install agent templates for Claude Code subagent feature

**Steps:**

1. Navigate to **My Settings → Integrations → Agent Templates**
2. Click "Generate Import Prompt"
3. System generates prompt with **tokenized download link** (15-min TTL)
4. Copy prompt and paste into Claude Code terminal
5. Claude Code downloads ZIP containing `*.md` agent template files
6. Templates installed to `~/.claude/agents/`
7. Restart Claude Code
8. Subagent templates available for orchestration

**Alternative:** Direct download link for manual installation

```
   ↓

   ↓
```

**Key Fields:**

- **`Project.name`** - Display name
- **`Project.description`** - Human-written requirements (INPUT for orchestrator)
- **`Project.mission`** - AI-generated plan (CREATED by orchestrator during staging)
- **`Project.context_budget`** - Maximum tokens for session management

**Distinction:**

- **Description** = What user wants (input)
- **Mission** = How orchestrator plans to achieve it (AI-generated output)

### 6.2 Activating a Project

```
1. Click "Activate" on project
   ↓
2. System checks if another project is active
   ↓
3. If yes: Deactivates current project (with warning)
   ↓
4. New project activated
   ↓
5. Unique dynamic link created: /projects/{project_id}?via=jobs
   ↓
6. "Active" badge on project card
   ↓
7. "Launch Project" button enabled
```

**Constraint:** Only ONE project can be active per product at any time.

### 6.3 Project States

- **inactive** - Created but not activated
- **active** - Currently being worked on (jobs can be created)
- **completed** - All jobs finished, project closed
- **cancelled** - Stopped before completion
- **deleted** - Soft deleted (10-day expiry, recoverable)

**Data Retention:** Deactivated/cancelled/completed projects keep ALL data (mission, agents, messages) for future reactivation.

## 9. Job Orchestration Workflow

### 9.1 Understanding the Workflow Phases

**PHASE 1: STAGING (Orchestrator Plans)**

- User activates project
- Orchestrator receives thin prompt
- Orchestrator fetches context from MCP server
- Orchestrator analyzes requirements
- Orchestrator creates mission plan
- Orchestrator selects agents and assigns jobs
- Mission and agent cards appear in UI

**PHASE 2: IMPLEMENTATION (Agents Execute)**

- User launches jobs
- Specialist agents receive thin prompts
- Agents fetch their missions from MCP server
- Agents execute assigned work
- Agents report progress via MCP tools
- Orchestrator coordinates via message queue

### 9.2 Job Staging Workflow (Detailed)

**Step 1: Navigate to Launch Panel**

```
1. Activate project
   ↓
2. Click "Launch Project" button OR navigate to Jobs sidebar link
   ↓
3. System loads: /projects/{project_id}?via=jobs
   ↓
4. Launch panel appears with "Launch" tab active
   ↓
5. UI shows:
   - Orchestrator card (empty, waiting)
   - Mission window (empty)
   - Project description (editable)
   - "Stage Project" button
```

**Step 2: Stage Project (Generate Thin Prompt)**

```
1. User clicks "Stage Project" button
   ↓
2. System calls: POST /api/prompts/orchestrator
   ↓
3. Backend creates MCPAgentJob record:
   - agent_type = "orchestrator"
   - status = "waiting"
   - job_id = new UUID
   - mission = (empty - will be populated by orchestrator)
   - job_metadata = {field_priorities, user_id}
   ↓
4. System generates THIN prompt (~10 lines):
   ```
   I am Orchestrator (instance 1) for Project "{project.name}".

   IDENTITY:
   - Orchestrator ID: {orch_id}
   - Project ID: {project_id}
   - Tenant: {tenant_key}

   INSTRUCTIONS:
   1. Fetch full context: get_orchestrator_instructions('{orch_id}', '{tenant_key}')
   2. Analyze requirements and create mission plan
   3. Select agents: spawn_agent_job()
   4. Persist mission: update_project_mission()

   Begin by fetching instructions.
   ```
   ↓
5. Prompt displayed in copyable modal
   ↓
6. User copies prompt
```

**Step 3: Execute Orchestrator (CLI Terminal)**

```
1. User pastes prompt into CLI terminal (Claude Code, Codex, or Gemini)
   ↓
2. Agent executes prompt
   ↓
3. Agent calls: get_orchestrator_instructions(orch_id, tenant_key)
   ↓
4. MCP tool returns:
   - Project.description (human requirements)
   - Product context (vision + tech stack - CONDENSED via field priorities)
   - Available agent templates
   - Token budget info
   - Estimated tokens: ~6000 (vs 30000 without thin client)
   ↓
5. Orchestrator analyzes requirements
   ↓
6. Orchestrator creates mission plan (condensed strategy)
   ↓
7. Orchestrator selects agents (e.g., Implementer, Tester, Reviewer)
   ↓
8. Orchestrator calls: spawn_agent_job() for each agent
   - Agent job created in database
   - Agent mission stored in MCPAgentJob.mission field
   - Thin prompt generated for agent (~10 lines)
   - WebSocket broadcast triggers UI update
   ↓
9. Orchestrator calls: update_project_mission(mission_plan)
   - Project.mission field updated
   - WebSocket broadcast triggers UI update
   ↓
10. UI updates automatically:
   - Mission appears in mission window
   - Agent cards appear in grid
   - "Launch Jobs" button enabled
```

**Key MCP Tools Used:**

- `get_orchestrator_instructions()` - Fetch condensed context
- `spawn_agent_job()` - Create agent job with thin prompt
- `update_project_mission()` - Persist mission to database

### 9.3 Job Implementation Workflow

**Step 1: Choose Implementation Mode**

```
1. User clicks "Launch Jobs" button
   ↓
2. System navigates to "Implementation" tab
   ↓
3. UI shows toggle: "Claude Code CLI" vs "Legacy CLI"
   ↓
4. User selects mode
```

**Mode A: Claude Code CLI (Single Terminal + Subagents)**

```
1. Toggle switched to "Claude Code CLI"
   ↓
2. Orchestrator prompt button enabled
   ↓
3. Individual agent prompt buttons DISABLED
   ↓
4. User clicks "Get Orchestrator Prompt"
   ↓
5. System generates orchestrator implementation prompt:
   ```
   I am Orchestrator for Project "{name}".

   MISSION:
   {Project.mission}

   AGENTS ASSIGNED:
   - Agent 1: {agent_type} (job_id: {id})
   - Agent 2: {agent_type} (job_id: {id})
   ...

   INSTRUCTIONS:
   Use Claude Code subagent system to spawn agents:
   - /spawn {agent_type_1} "Fetch mission: get_agent_mission('{job_id_1}', '{tenant_key}')"
   - /spawn {agent_type_2} "Fetch mission: get_agent_mission('{job_id_2}', '{tenant_key}')"

   Coordinate agent work and track progress.
   ```
   ↓
6. User copies prompt and pastes in single terminal
   ↓
7. Orchestrator uses Claude Code's native `/spawn` command
   ↓
8. Subagents spawned automatically
   ↓
9. Each subagent calls: get_agent_mission(agent_job_id, tenant_key)
   ↓
10. Subagent receives full mission from database
   ↓
11. Subagents execute work, orchestrator coordinates
```

**Mode B: Legacy CLI (Multiple Terminals)**

```
1. Toggle remains on "Legacy CLI" (default)
   ↓
2. Orchestrator prompt button enabled
   ↓
3. Individual agent prompt buttons enabled
   ↓
4. User copies orchestrator prompt first
   ↓
5. Paste in Terminal 1 (orchestrator determines parallel vs sequential)
   ↓
6. User copies each agent prompt
   ↓
7. Paste in separate terminals (Terminal 2, 3, 4, etc.)
   ↓
8. Each agent calls: get_agent_mission(agent_job_id, tenant_key)
   ↓
9. Each agent receives full mission from database
   ↓
10. Agents execute work independently
   ↓
11. Orchestrator coordinates via message queue
```

**Step 2: Agent Execution Loop**

```
AGENT WORKING:
1. Agent fetches mission: get_agent_mission(job_id, tenant_key)
   ↓
2. Agent executes assigned work (write code, run tests, etc.)
   ↓
3. Agent reports progress: report_progress(job_id, completed_todo, files_modified, context_used, tenant_key)
   ↓
4. Progress stored in message queue
   ↓
5. WebSocket broadcast updates UI
   ↓
6. Agent checks for messages: get_next_instruction(job_id, agent_type, tenant_key)
   ↓
7. If orchestrator sent feedback → Agent processes it
   ↓
8. If context limit approaching → Agent prepares handoff
   ↓
9. Agent completes work: complete_job(job_id, result, tenant_key)
   ↓
10. Job marked "complete" in database
   ↓
11. WebSocket broadcast updates UI
```

**Step 3: User Interaction During Jobs**

```
USER CAN:
- View real-time progress in Implementation tab
- Send messages to specific agents via message pane
- Broadcast message to all agents
- Monitor agent status (waiting, working, blocked, complete, failed)
- Cancel jobs if needed (graceful cancellation)

AGENTS RESPOND:
- Via MCP tool calls (stored in message queue)
- Via terminal output (visible to user in CLI)
```

### 9.4 Project Closeout

```
1. All agents mark jobs complete
   ↓
2. Orchestrator detects all jobs done
   ↓
3. Orchestrator generates closeout summary
   ↓
4. Orchestrator creates closeout prompt (git commands)
   ↓
5. User executes closeout (commit, push, tag)
   ↓
6. User clicks "Complete Project" in UI
   ↓
7. Project.status = "completed"
   ↓
8. Agent jobs decommissioned (status = "decommissioned")
   ↓
9. Project appears in completed projects list
   ↓
10. Project can be reactivated if needed (agents restored)
```

## 10. Agent Communication & Messaging

### 10.1 Message Queue Architecture

**Database Table:** `messages` (with `agent_communication_queue` compatibility layer)

**Message Flow:**

```
Agent → MCP Tool → Message Queue → Database → WebSocket → UI
                                    ↓
                              Other Agents
```

### 10.2 Message Types

1. **progress** - Incremental work updates
2. **user_feedback** - User messages to agents
3. **orchestrator_instruction** - Orchestrator commands
4. **handoff_request** - Context handoff signals
5. **context_warning** - Approaching token limits
6. **error** - Error reports

### 10.3 Sending Messages (User → Agent)

```
1. User clicks message icon on agent card
   ↓
2. Message compose dialog appears
   ↓
3. User types message
   ↓
4. User clicks "Send" or "Broadcast to All"
   ↓
5. System calls: send_message(job_id, to_agent, message, tenant_key, priority)
   ↓
6. Message stored in queue
   ↓
7. Agent fetches via: get_next_instruction(job_id, agent_type, tenant_key)
   ↓
8. Agent processes message and responds
```

### 10.4 Agent Progress Reporting

```
Agent calls: report_progress(
  job_id,
  completed_todo,
  files_modified,
  context_used,
  tenant_key
)
   ↓
MCP tool stores in message queue
   ↓
WebSocket broadcast to UI
   ↓
Progress bar updates
   ↓
If context_used > 25K tokens → Warning issued
If context_used > 28K tokens → Handoff recommended
If context_used > 29K tokens → CRITICAL handoff required
```

## 11. Context Management & Perpetual Operation

### 11.1 The Perpetual Context Problem

**Challenge:** AI tools have token limits (~200K for Claude, ~32K for older models)

**Solution:** GiljoAI manages perpetual context through:

1. **Database-driven instructions** - Full context not in prompts
2. **Thin client architecture** - 70% token reduction ACTIVE
3. **Field priority system** - User controls what context is included
4. **Orchestrator succession** - Auto-handoff at 90% capacity
5. **Message queue** - Async communication doesn't consume context

### 11.2 Field Priority Configuration

```
User Settings → Context Priority
   ↓
User drags fields to set order:
1. Vision Document (highest priority)
2. Tech Stack
3. Product Description
4. Project Description
5. Agent Templates
6. Previous Messages (lowest priority)
   ↓
System includes fields top-to-bottom until token budget (2000) reached
   ↓
Priorities stored in Settings table
   ↓
Used when generating missions for agents
```

**Benefit:** User controls what context agents receive

### 11.3 Orchestrator Succession Workflow

```
1. Orchestrator tracks context usage: context_used / context_budget
   ↓
2. At 90% capacity (135K / 150K tokens):
   - Orchestrator triggers succession
   ↓
3. Orchestrator calls: spawn_succession_orchestrator()
   ↓
4. New orchestrator created with:
   - instance_number = old_instance + 1
   - spawned_by = old_orchestrator_id
   - handover_summary = condensed state (<10K tokens)
   ↓
5. Old orchestrator decommissioned (read-only advisor)
   ↓
6. New orchestrator continues work with fresh context window
   ↓
7. Lineage preserved: instance 1 → instance 2 → instance 3 → ...
```

**Manual Succession:** User can trigger via `/gil_handover` slash command or UI "Hand Over" button

## 12. MCP Tools Reference

### 12.1 Orchestrator Tools

**`get_orchestrator_instructions(orchestrator_id, tenant_key)`**

- **Purpose:** Fetch condensed context for mission planning
- **Returns:** Project description, product context (condensed), agent templates, token budget
- **Token Reduction:** ~6000 tokens (vs ~30000 without thin client)

**`update_project_mission(project_id, mission, tenant_key)`**

- **Purpose:** Persist AI-generated mission to database
- **Effect:** Mission appears in UI, agents can reference it

**`spawn_agent_job(agent_type, agent_name, mission, project_id, tenant_key, parent_job_id)`**

- **Purpose:** Create specialist agent job for execution
- **Returns:** agent_job_id, thin prompt (~10 lines), token estimates
- **Effect:** Agent job created, WebSocket broadcast updates UI

**`orchestrate_project(project_id, tenant_key)`**

- **Purpose:** Full orchestration pipeline (rarely used - prefer thin client)
- **Effect:** Vision processing, mission generation, agent selection, job spawning

### 12.2 Agent Execution Tools

**`get_agent_mission(agent_job_id, tenant_key)`**

- **Purpose:** Fetch agent-specific mission from database
- **Returns:** Mission, project context, agent identity
- **Token Estimate:** ~2000 tokens (mission only)

**`acknowledge_job(job_id, agent_id, tenant_key)`**

- **Purpose:** Claim job (pending → active transition)
- **Effect:** Job started, timestamp recorded

**`report_progress(job_id, completed_todo, files_modified, context_used, tenant_key)`**

- **Purpose:** Report incremental work progress
- **Effect:** Message stored, UI updated, context warnings issued if needed

### 13.1 How Instructions Are Stored

**Key Tables:**

```sql
-- Orchestrator context (INPUT for planning)
SELECT description FROM projects WHERE id = {project_id};

-- Generated mission (OUTPUT from orchestrator)
SELECT mission FROM projects WHERE id = {project_id};

-- Agent-specific instructions
SELECT mission FROM mcp_agent_jobs WHERE job_id = {agent_job_id};

SELECT name, role, description FROM agent_templates
WHERE tenant_key = {tenant_key} AND is_active = true;
```

### 13.2 Thin Prompt vs Full Context

**Thin Prompt (what user copies):**

```
I am Backend Agent for Project "API Refactor".

FETCH MISSION:
get_agent_mission('{job_id}', '{tenant_key}')

Execute and report progress.
```

**Full Context (fetched from database):**

```
AGENT MISSION (from MCPAgentJob.mission field):
Refactor authentication endpoints to use JWT tokens.

Requirements:
- Update /api/auth/login to return JWT
- Add JWT verification middleware
- Update user model with refresh tokens
- Write integration tests

Tech Stack:
- Python 3.11, FastAPI
- PostgreSQL for token storage
- pytest for testing

CONTEXT (from Product.config_data):
- Architecture: Microservices
- Auth Strategy: JWT with refresh tokens
- Test Coverage Target: >80%

VISION (from VisionDocument.content - chunked):
Chunk 1: Product vision emphasizes security...
Chunk 2: API design follows RESTful patterns...

Estimated tokens: 2000
```

### 13.3 Token Reduction Mechanics

**Without Thin Client (OLD):**

- Prompt size: 30,000 tokens (full vision embedded)
- Context consumed: Immediate
- Scaling: Cannot handle large products

**With Thin Client (NEW - ACTIVE):**

- Prompt size: 50 tokens (thin prompt only)
- Context consumed: 6,000 tokens (condensed via field priorities)
- Scaling: Unlimited product size
- **Reduction: 70%** (6000 / 30000 = 30% of original = 70% reduction)

### 13.4 Field Priority Token Management

```
User Priority Order:
1. Vision (Priority 1) → 1500 tokens
2. Tech Stack (Priority 2) → 300 tokens
3. Product Desc (Priority 3) → 100 tokens
4. Project Desc (Priority 4) → 50 tokens
5. Agent Templates (Priority 5) → 50 tokens
   ──────────────────────────────
   Total: 2000 tokens (within budget)

Priority 6+ fields: Excluded (budget exhausted)
```

**Result:** Orchestrator receives exactly what user prioritizes, nothing more.

## 14. Key Workflow Clarifications

### 14.1 Static vs Dynamic Prompts

**STATIC (Easy Copy-Paste):**

- MCP CLI setup (attach server to tool)
- Slash command installation
- Agent template download (Claude Code)
- These are ONE-TIME setup steps

**DYNAMIC (Database-Driven):**

- Orchestrator staging prompts (fetch context from database)
- Agent execution prompts (fetch missions from database)
- These are PER-PROJECT, generated on demand

### 14.2 Staging vs Implementing

**STAGING (Orchestrator Plans):**

- Orchestrator reads `Project.description`
- Orchestrator fetches product context via MCP
- Orchestrator creates mission plan
- Orchestrator spawns agent jobs
- **Orchestrator does NOT execute work**

**IMPLEMENTING (Agents Execute):**

- Specialist agents fetch missions via MCP
- Agents EXECUTE assigned work (write code, run tests)
- Agents report progress via MCP
- Orchestrator coordinates via messages
- **Agents DO the actual work**

### 14.3 Description vs Mission

**`Project.description`** (Human-Written Input):

```
Create a REST API for user authentication with JWT tokens.
Include login, logout, refresh token rotation, and password reset.
```

**`Project.mission`** (AI-Generated Plan):

```
MISSION PLAN:

Phase 1: Database Schema (Backend Agent)
- Add refresh_tokens table
- Update users table with token fields

Phase 2: Authentication Endpoints (Backend Agent)
- /api/auth/login (POST)
- /api/auth/logout (POST)
- /api/auth/refresh (POST)
- /api/auth/reset-password (POST)

Phase 3: Middleware (Backend Agent)
- JWT verification decorator
- Token expiry handling

Phase 4: Testing (Tester Agent)
- Integration tests for all endpoints
- Test token rotation logic
- Test password reset flow

Phase 5: Documentation (Documenter Agent)
- API documentation
- Authentication flow diagrams
```

**Key Difference:**

- Description = WHAT (input)
- Mission = HOW (output)

## 16. Appendix: Status States

### Project Status

- **inactive** - Created but not activated
- **active** - Currently being worked on
- **completed** - All jobs finished
- **cancelled** - Stopped before completion
- **deleted** - Soft deleted (10-day expiry)

### Agent Job Status

- **waiting** - Created, not acknowledged
- **working** - Agent actively executing
- **blocked** - Waiting for dependency
- **complete** - Successfully finished
- **failed** - Terminated with error
- **cancelled** - User-requested cancellation
- **decommissioned** - Project closed, job archived

### Message Status

- **pending** - Not yet read by recipient
- **acknowledged** - Read but not acted upon
- **completed** - Action taken
- **failed** - Delivery or processing error
