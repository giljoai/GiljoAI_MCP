## Current Architecture

### Baseline Migration (f504ea46e988)

**What It Does**:
- Creates ALL 32 database tables in single atomic transaction
- Installs pg_trgm extension for full-text search
- Generated from SQLAlchemy models (not manual SQL)
- Completes in <1 second on fresh installations

**Tables Created** (32 total):
```
Authentication:  users, api_keys, mcp_sessions
Products:        products, vision_documents, visions, git_configs, git_commits
Projects:        projects, sessions, discovery_config
Agents:          mcp_agent_jobs, agent_interactions, jobs, messages
Templates:       agent_templates, template_archives, template_augmentations, template_usage_stats
Tasks:           tasks
Context:         context_index, large_document_index, mcp_context_index, mcp_context_summary
Configuration:   configurations, settings, setup_state, optimization_rules, optimization_metrics
Utilities:       api_metrics, download_tokens
Alembic:         alembic_version
```

### Why Nuclear Reset?

**Problem** (Old 44-Migration Chain):
- Chicken-and-egg dependency conflicts (tables referenced before creation)
- Fresh installs FAILED with "table not found" errors
- 5+ minute migration time (if it worked at all)
- Complex debugging required for every change

**Solution** (Single Baseline):
- Zero dependency conflicts (all tables created together)
- Fresh installs SUCCESS in 0.57 seconds
- Clean foundation for future incremental migrations
- Simple, reliable, production-ready

```bash
# 1. Create database

# 2. Set DATABASE_URL environment variable
export DATABASE_URL="postgresql://postgres:password@localhost:5432/giljo_mcp"

# 3. Run migration

# Output: Running upgrade  -> f504ea46e988, baseline_schema_all_27_tables
# Time: ~0.57 seconds
# Result: 32 tables created + pg_trgm extension installed

# 4. Verify installation
psql -U postgres -d giljo_mcp -c "\dt"
# Should list 32 tables
```

## Creating New Migrations

### When to Create a Migration

Create a new migration when you need to:
- Add a new table
- Add a new column to existing table
- Create a new index
- Change column type or constraint
- Add database-level validation
- Modify foreign key relationships

### Migration Creation Workflow

```bash
# 1. Update SQLAlchemy Models
# Edit files in src/giljo_mcp/models/
# Example: Add new column to Product model

# 2. Generate Migration
alembic revision --autogenerate -m "add_feature_description"
# Output: Generating migrations/versions/<revision_id>_add_feature_description.py

# 3. Review Generated Migration
# IMPORTANT: Alembic autogenerate can miss things!
# - Review the generated file carefully
# - Check for missing indexes
# - Verify foreign key constraints
# - Add data migrations if needed

# 4. Test Migration on Dev Database
alembic upgrade head
# Verify tables/columns created correctly

# 5. Test Rollback
alembic downgrade -1
# Verify rollback works cleanly

# 6. Test Upgrade Again
alembic upgrade head
# Confirm migration is idempotent

# 7. Commit Migration File
git add migrations/versions/<revision_id>_*.py
git commit -m "feat: Add feature description (migration)"
```

### Migration Best Practices

**DO**:
- Always test on dev database first
- Review autogenerated migrations carefully
- Test both upgrade AND downgrade
- Include data migrations when needed
- Use descriptive migration messages
- Document breaking changes in migration file

**DON'T**:
- Manually edit existing migrations (creates version conflicts)
- Skip testing rollback (downgrade must work)
- Commit without testing (broken migrations block installs)
- Use raw SQL without Alembic's op.execute() wrapper
- Forget to add migration file to git

## Multi-Tenant Safety

### CRITICAL Requirements

ALL new tables MUST include:

```python
class MyNewTable(Base):
    __tablename__ = "my_new_table"

    id = Column(String(36), primary_key=True, default=generate_uuid)
    tenant_key = Column(String(36), nullable=False, index=True)  # REQUIRED
    name = Column(String(100), nullable=False)

    __table_args__ = (
        Index("idx_my_table_tenant", "tenant_key"),  # REQUIRED
        # Compound indexes should have tenant_key first for query performance
        Index("idx_my_table_tenant_status", "tenant_key", "status"),
    )
```

**Why This Matters**:
- Multi-tenant isolation prevents data leakage
- Index on tenant_key ensures fast queries
- Compound indexes with tenant_key first optimize filtering

### Tenant Isolation Verification

```sql
-- Verify all tables have tenant_key column
SELECT table_name
FROM information_schema.columns
WHERE table_schema = 'public'
  AND column_name = 'tenant_key'
ORDER BY table_name;

-- Should return 28+ tables (excluding alembic_version and some utility tables)
```

## Common Migration Patterns

### Adding a New Column

```python
def upgrade():
    op.add_column('products',
        sa.Column('new_field', sa.String(255), nullable=True))

def downgrade():
    op.drop_column('products', 'new_field')
```

### Adding an Index

```python
def upgrade():
    op.create_index('idx_products_new_field', 'products', ['new_field'])

def downgrade():
    op.drop_index('idx_products_new_field', 'products')
```

### Adding a Foreign Key

```python
def upgrade():
    op.add_column('tasks',
        sa.Column('project_id', sa.String(36), nullable=True))
    op.create_foreign_key(
        'fk_tasks_project_id',
        'tasks', 'projects',
        ['project_id'], ['id'],
        ondelete='CASCADE'
    )

def downgrade():
    op.drop_constraint('fk_tasks_project_id', 'tasks', type_='foreignkey')
    op.drop_column('tasks', 'project_id')
```

### Data Migration with Backfill

```python
from sqlalchemy import text

def upgrade():
    # Add column (nullable first)
    op.add_column('users',
        sa.Column('role', sa.String(32), nullable=True))

    # Backfill existing rows
    conn = op.get_bind()
    conn.execute(text("""
        UPDATE users
        SET role = 'user'
        WHERE role IS NULL
    """))

    # Make column NOT NULL after backfill
    op.alter_column('users', 'role',
        existing_type=sa.String(32),
        nullable=False)

def downgrade():
    op.drop_column('users', 'role')
```

## Troubleshooting

### Migration Fails with "table already exists"

**Cause**: You may have run `Base.metadata.create_all()` manually
**Fix**: Drop the database and run migrations cleanly

```bash
psql -U postgres -c "DROP DATABASE giljo_mcp;"
psql -U postgres -c "CREATE DATABASE giljo_mcp;"
alembic upgrade head
```

### Migration Fails with "column already exists"

**Cause**: Migration was partially applied
**Fix**: Check current migration version and manually fix

```bash
# Check current version
alembic current
# Output: <revision_id> (head)

# Check migration history
alembic history
# Shows all migrations and current position
```

### Rollback Failed

**Cause**: Downgrade logic may have errors
**Fix**: Restore from backup, fix downgrade, test again

```bash
# Restore database from backup
psql -U postgres -d giljo_mcp < backups/giljo_mcp_YYYYMMDD.sql

# Fix downgrade function in migration file
# Test downgrade
alembic downgrade -1
```

### For Fresh Installations

```bash
# 1. Database already created by install.py
# 2. Baseline migration automatically run by install.py
# 3. No manual intervention needed
```

### For Schema Updates (Future)

When we have production data and need schema changes:

```bash
# 1. ALWAYS backup production database first
pg_dump -U postgres -d giljo_mcp -F p -f backups/giljo_mcp_$(date +%Y%m%d_%H%M%S).sql

# 2. Test migration on staging database (with production data copy)
alembic upgrade head

# 3. Verify application works with new schema
pytest tests/

# 4. Apply to production (during maintenance window)
alembic upgrade head

# 5. Monitor application logs for errors
tail -f logs/api.log
```
