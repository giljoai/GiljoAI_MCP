# Chapter 2: Context Management Philosophy

Core Philosophy: Intelligent Context Assembly Through Two-Phase Knowledge Retrieval
Giljo implements a sophisticated context management system that balances comprehensive knowledge access with strict token efficiency. Rather than searching on every query, Giljo uses a two-tier embedding architecture to enable ambient project awareness while delivering complete, trustworthy information when deeper knowledge is required.
The Critical Architecture Decision: Routing vs. Response
The Problem We Solved
Early implementations faced a fundamental tension: descriptions helped find relevant documents, but the system would present descriptions as answers rather than using them to guide retrieval of actual content. This created incomplete responses that undermined professional trust.
The Solution: Two-Phase Retrieval Architecture
Phase 1: Discovery via Descriptions
• Human-provided or AI-enhanced descriptions act as routing layers
• Descriptions use chunk_index = -1 to appear first in similarity searches
• Purpose: Identify which documents contain relevant information
• Result: Document-level relevance without content details
Phase 2: Complete Content Delivery
• Once relevant documents are identified, ALL chunks from those documents are included
• Content chunks use ascending indices (0, 1, 2...) for proper ordering
• Purpose: Provide complete information for accurate reasoning
• Result: LLM sees full context, not just descriptions

Implementation Logic
Technical Implementation Note: Description chunks use chunk_index = -1 for discovery priority, content chunks use ascending indices (0, 1, 2...) for response material. The system performs two-phase retrieval: descriptions identify relevant documents, then ALL chunks from those documents are included in context for complete information delivery.
Why This Matters for Professionals:
• Lawyers get complete contract language, not just summaries
• Financial analysts see full data tables, not just descriptions
• Consultants access entire reports, not just abstracts
• Trust is built through transparency and completeness

Two-Tier Embedding Architecture
Document Fingerprints: The Routing Layer
Purpose: Enable ambient project awareness and act as intelligent routing for content discovery
Implementation:
• One summary embedding per document stored in documents.fingerprint_embedding
• Generated from user descriptions (enhanced by AI when needed)
• Used for fast document-level relevance checking
• Enables "awareness" without full search overhead

Storage Strategy:
sqldocuments table:
- user_description (what user provided)
- generated_summary (AI-enhanced or auto-generated)
- fingerprint_embedding (vector of the summary)
Routing Logic: Query → Fingerprint similarity → Document selection → Full chunk retrieval
Document Chunks: The Content Layer
Purpose: Detailed knowledge retrieval with complete information preservation
Size: 1600-2000 characters with 200-300 character overlap
Storage: Full embeddings in document_chunks table with proper chunk ordering
Usage: Deep search only on documents identified as relevant by fingerprint matching

Chunk Ordering Strategy:
chunk_index = -1: Description/summary chunks (for routing)
chunk_index = 0, 1, 2...: Actual content chunks (for response)
System ensures content appears after descriptions in context assembly

python
class ModelManager:
    def __init__(self):
        self.vllm_process = None
        self.current_model = None
        self.model_directory = "/models"
    
    async def switch_model(self, new_model: str):
        # Graceful shutdown of current model
        if self.vllm_process:
            await self.shutdown_current_model()
        
        # Start new vLLM instance
        self.vllm_process = await self.start_vllm_server(
            model=new_model,
            port=8000,
            max_model_len=4096,
            dtype="float16"
        )
        
        self.current_model = new_model
        return {"status": "loaded", "model": new_model}

pythonclass DocumentQueue:
    def estimate_wait_time(self, doc):
        base_estimates = {
            'simple_pdf': 30,      # seconds
            'complex_pdf': 120,    
            'powerpoint': 180,
            'scanned_doc': 240
        }
        
        position = self.get_queue_position(doc)
        wait_time = sum([base_estimates[d.type] for d in self.queue[:position]])
        
        if wait_time > 180:
            return f"Complex document processing: ~{wait_time//60} minutes. Grab a coffee! ☕"
        else:
            return f"Processing: ~{wait_time} seconds"

Dynamic Loading Strategy

pythonclass ModelManager:
    def __init__(self, vram_limit=20):
        self.vram_limit = vram_limit
        self.loaded_models = {
            'instructor_xl': self.load_instructor(),  # Always loaded
        }
        self.usage_stats = defaultdict(int)
    
    def load_for_document(self, doc_type):
        if doc_type.needs_ocr and 'easyocr' not in self.loaded_models:
            self.load_model('easyocr')
        
        if doc_type.needs_visual and 'florence_2' not in self.loaded_models:
            self.ensure_vram_available(2.5)
            self.load_model('florence_2')
                
    def load_instructor(self):
        from InstructorEmbedding import INSTRUCTOR
        model = INSTRUCTOR('hkunlp/instructor-xl')
        model.cuda(self.doc_gpu_id)
        return model
