Chapter 5: Multi-Project File Association & Lifecycle Management
Core Philosophy: Organizational Knowledge as a Living Asset
GIljo implements a sophisticated file management architecture that mirrors how knowledge naturally flows within organizations. Files serve multiple projects, evolve through versions, and accumulate institutional memory over time. Rather than forcing rigid ownership models or destructive deletion patterns, the system embraces a lifecycle approach where knowledge matures from active use through archival to eventual retirement - always under user control, always recoverable until explicitly purged.
This philosophy acknowledges that SMB organizations rarely need to truly delete information. What they need is intelligent organization that keeps current work accessible while preserving historical knowledge for compliance, reference, and learning.
The Evolution from Single to Multi-Project Association
Initial Constraints
The original implementation restricted files to single project assignment, creating artificial barriers that don't reflect real work patterns:

Duplicate uploads of the same template across projects
Version confusion when similar documents existed in isolation
Wasted processing and storage for redundant files
Broken knowledge connections between related initiatives

The Multi-Project Breakthrough
Recognizing that SMB users - particularly in legal and financial services - routinely reference documents across multiple projects, GIljo now supports true multi-project file association. A contract template can simultaneously serve the M&A project, the compliance review, and the training materials without duplication or confusion.

CREATE TABLE project_documents (
    project_id UUID REFERENCES projects(id),
    document_id UUID REFERENCES documents(id),
    linked_date TIMESTAMP DEFAULT NOW(),

-- Knowledge preservation with lifecycle awareness
CREATE TABLE document_chunks (
    id UUID PRIMARY KEY,
    document_id UUID REFERENCES documents(id),

    source_deleted_at TIMESTAMP,
    original_filename VARCHAR(255),
    original_uploader_id VARCHAR(255)
);

File Immutability: Once uploaded, files are never modified - only replaced by new versions. This ensures consistency across projects and maintains clear audit trails.
Lifecycle-Aware Storage: Files progress through active → archived → deleted states, with knowledge preservation at each stage.
Global Existence: Files exist at the organizational level, with projects maintaining references rather than ownership.

Search Weight Management
pythondef calculate_search_relevance(self, chunk, base_similarity):
    """Apply lifecycle-aware relevance scoring"""
    
    # Get document status
    doc_status = self.get_document_status(chunk.document_id)
    
    # Apply weight based on lifecycle stage
    if doc_status == 'archived':
        # Drastically reduce relevance for archived content
        return base_similarity * 0.1
    elif chunk.source_deleted_at is not None:
        # File deleted but knowledge preserved
        return base_similarity * 0.05
    else:
        # Active file - full relevance
        return base_similarity * 1.0

Search Integration
sql-- Modified search query with lifecycle awareness
WITH weighted_search AS (
    SELECT 
        dc.content,
        dc.chunk_index,
        d.filename,
        d.status,
        -- Apply lifecycle weighting
        CASE 
            WHEN d.status = 'archived' THEN 
                (1 - (dc.embedding <=> $1)) * 0.1
            WHEN dc.source_deleted_at IS NOT NULL THEN 
                (1 - (dc.embedding <=> $1)) * 0.05
            ELSE 
                (1 - (dc.embedding <=> $1))
        END as weighted_similarity
    FROM document_chunks dc
    JOIN documents d ON dc.document_id = d.id
    WHERE project_id = $2
)
SELECT * FROM weighted_search
ORDER BY weighted_similarity DESC
LIMIT 20;
Auto-Archive Suggestions
pythonclass ArchiveAdvisor:
    async def suggest_archives(self, days_inactive=365):
        """Suggest files for archival based on usage patterns"""
        
        suggestions = await self.db.fetch("""
            SELECT d.*, 
                   MAX(pd.linked_date) as last_used,
                   COUNT(DISTINCT pd.project_id) as project_count
            FROM documents d
            LEFT JOIN project_documents pd ON d.id = pd.document_id
            WHERE d.status = 'active'
            GROUP BY d.id
            HAVING MAX(pd.linked_date) < NOW() - INTERVAL '%s days'
                OR MAX(pd.linked_date) IS NULL
            ORDER BY last_used ASC
        """, days_inactive)
        
        return suggestions

Conclusion
This comprehensive lifecycle approach transforms file management from a binary keep/delete decision into an intelligent progression that mirrors how organizations naturally handle information. By making archival the default "deletion" behavior, GIljo eliminates the anxiety of losing important data while keeping active work uncluttered.
The multi-project association capability combined with lifecycle management creates a unique value proposition: organizations can freely share knowledge across initiatives while maintaining clean, focused workspaces. The system grows more valuable over time as archived knowledge accumulates, providing a searchable institutional memory that informs without overwhelming.
Most importantly, this approach acknowledges that in professional services, information rarely becomes worthless - it simply becomes less immediately relevant. By preserving everything while intelligently managing visibility and search relevance, GIljo delivers both the clean workspace users want and the comprehensive knowledge preservation their business requires.
