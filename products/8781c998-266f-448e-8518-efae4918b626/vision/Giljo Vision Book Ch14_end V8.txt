#1 Technical Debt: Background Processing Architecture Decision
Context & Logical Flow
Problem Identification:
Document comparison identified as "deal breaker" feature for SMB market
Real-world power user workflow: Compare → Harmonize → Merge documents
Long document processing (3-8 minutes) conflicts with team collaboration
Single vLLM model serves all users - intensive tasks block others
Strategic Analysis:
Hardware specs (GIljo 5: 20GB, GIljo 10/15: 24GB) can handle document comparison
Progressive document analysis feasible with chunking strategies
User preference: "Accuracy over speed" - willing to wait for quality results
SMB team dynamics: Power users shouldn't block casual users
Decision Made: Background Processing Architecture for both document ingestion AND long chat generations
Technical Architecture Decisions
Background Processing Scope:
Document uploads (initial processing/chunking/embedding)
Document comparison and analysis
Long-form generation (harmonization, merging, research mode)
Any task estimated >2 minutes processing time
Queue Priority System:
Interactive chat: Immediate priority
Background tasks: Lower priority, interruptible
Progress notifications: 25%, 50%, 75% completion
Smart interruption: "Busy working on prior request" messaging
User Experience Pattern:
Submit Task → Background Processing → Continue Normal Work → Completion Notification

#2 Technical Debt: Conversational Token Budget Management
Context & Problem Statement
Token Budget Reality:

Designed 4000 token budget: System (100) + Project (100) + User (50) + Chat (500) + Vector (3250)
Vector search frequently returns 4,800+ tokens from 15+ relevant chunks
Current system has "token budgeting designed but not implemented"
Risk of prompt complexity creep with hard cutoffs or dynamic compression

User Experience Problem:

Hard cutoffs miss critical context ranked lower
Dynamic compression loses important details
Complex scoring algorithms create maintenance burden
Pop-ups and UI controls break conversational flow

Solution: Conversational Token Continuation
Natural Language Budget Management:
Instead of complex algorithms, use conversational interaction to handle overflow:
User: "What are the key risks in our Q4 contracts?"

GIljo: "Based on my analysis of your contract portfolio, I found 3 primary risks:
[Analysis based on first 3,250 tokens of top-ranked chunks]

💡 I found additional relevant material that might contain more risk analysis. Would you like me to review the remaining material for a more comprehensive assessment?"

User: "Yes, continue"
GIljo: [Processes next 3,250 token batch with accumulated context]
Technical Implementation Strategy
Token Budget Logic:

Process top-ranked chunks within 3,250 token limit
Detect remaining high-relevance chunks above threshold
Offer natural continuation with context preservation
Maintain conversation state across continuation requests

Natural Language Patterns:

"I see there's more material that might be relevant..."
"There are additional [specific source types] I haven't reviewed yet..."
"Want me to dig deeper into [specific area]?"
Proactive suggestions: "I found 6 more email threads and 3 policy documents - want me to check the emails first?"
