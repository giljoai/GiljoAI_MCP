Chapter 7: RAG Optimization & Natural Language Document Interaction
Core Philosophy: Intelligent Document Awareness Through Natural Language
Giljo's RAG optimization strategy transcends traditional vector search by understanding how professionals naturally reference their documents. Rather than forcing users to learn special syntax or navigate complex query builders, the system combines agentic routing intelligence with natural language understanding to deliver precise results within project boundaries while enabling controlled cross-project insights.
This approach maintains the project-centric philosophy established in previous chapters while delivering the document intelligence users intuitively expect from modern AI systems.
The Natural Language Challenge: User Behavior vs System Capabilities
Real User Behavior Patterns
Analysis of natural user interactions reveals consistent patterns:

"Working on the volvo" (referring to volvo.txt)
"John needs to fix the motorcycle" (motorcycle.txt context)
"What car?" (expecting vehicle-related document awareness)
"Did I pay my loans with the credit card?" (expecting cross-project intelligence)
"In the budget proposal, what was our Germany conclusion?" (expecting cross-document analysis)

Traditional RAG Limitations
Standard vector search treats all chunks equally, missing the natural document-level associations users expect. This creates a fundamental disconnect between user mental models and system capabilities, leading to responses that feel "dumb" about uploaded documents.
Two-Tier Document Architecture: Working Documents vs Knowledge Base
The Fundamental Distinction
Giljo recognizes two different document relationships that serve distinct purposes:
Working Documents (Project-Attached)

Purpose: Active editing, analysis, and rewriting
Scope: Limited to current project context
Interaction: Direct manipulation and transformation
Examples: Contract being negotiated, report being written

Knowledge Base (Organization-Wide)

Purpose: Reference and institutional memory
Scope: Searchable across projects (with toggle)
Interaction: Read-only retrieval
Examples: Past contracts, completed reports, policies

Document Fingerprints: The Routing Layer
Using Instructor-XL with domain-specific instructions:
pythonclass DocumentFingerprinter:
    def __init__(self):
        self.embedder = InstructorXL()
        self.domain_instructions = {
            'legal': "Represent this legal document for case research:",
            'financial': "Represent this financial document for analysis:",
            'general': "Represent this business document for retrieval:"
        }
    
    def create_fingerprint(self, document, domain='general'):
        # Generate comprehensive summary
        summary = self.generate_summary(document)
        
        # Extract key entities
        entities = self.extract_entities(document)
        
        # Create fingerprint with domain instruction
        instruction = self.domain_instructions.get(domain, self.domain_instructions['general'])
        fingerprint_text = f"{summary}\nKey entities: {', '.join(entities)}"
        
        embedding = self.embedder.encode(instruction, fingerprint_text)
        
        return {
            'embedding': embedding,
            'summary': summary,
            'entities': entities,
            'domain': domain
        }

Conversational Document Memory
Implementation of context tracking across chat messages:
pythonclass ConversationContext:
    def __init__(self, retention_window=3):
        self.retention_window = retention_window
        
    def track_document_focus(self, chat_history, current_query):
        """Maintain awareness of recently discussed documents"""
        recent_messages = chat_history[-self.retention_window:]
        
        # Extract document mentions from recent context
        recent_document_mentions = []
        for message in recent_messages:
            entities = self.extract_entities_from_message(message)
            doc_refs = self.extract_document_references(message)
            recent_document_mentions.extend(entities + doc_refs)
        
        # Weight current query based on recent context
        context_weights = self.calculate_context_weights(
            recent_document_mentions, current_query
        )
        
        return context_weights
    
    def detect_contextual_references(self, query):
        """Detect references like 'the car', 'that document'"""
        contextual_patterns = {
            r'\bthe (car|vehicle|auto)\b': 'vehicle_type',
            r'\bthe (bike|motorcycle)\b': 'motorcycle_type',
            r'\bthat (document|file|report)\b': 'recent_document',
            r'\bthe (proposal|budget|plan)\b': 'document_type',
            r'\b(it|this|that)\b': 'pronoun_reference'
        }
        
        matches = []
        for pattern, ref_type in contextual_patterns.items():
            if re.search(pattern, query.lower()):
                matches.append({
                    'type': ref_type,
                    'pattern': pattern,
                    'requires_context': True
                })
        
        return matches

Entity Recognition and Knowledge Graphs
Entity Extraction Pipeline
pythonclass EntityExtractor:
    def __init__(self):
        self.patterns = {
            'company': r'\b[A-Z][A-Za-z\s&]+(?:Inc|LLC|Corp|Bank|LLP)\b',
            'money': r'\$[\d,]+\.?\d*',
            'date': r'\b(?:Jan|Feb|Mar|Apr|May|Jun|Jul|Aug|Sep|Oct|Nov|Dec)\s+\d{1,2}',
            'account': r'\b\d{4,}-?\d{4,}\b'
        }
    
    def extract_from_document(self, document):
        entities = {
            'companies': [],
            'amounts': [],
            'dates': [],
            'accounts': []
        }
        
        # Extract using patterns
        for entity_type, pattern in self.patterns.items():
            matches = re.findall(pattern, document.text)
            entities[entity_type].extend(matches)
        
        # Store in knowledge graph
        self.update_knowledge_graph(document.id, entities)
        
        return entities

Response Formatting with Document Attribution
Natural language response assembly with clear source attribution:
pythonclass NaturalLanguageResponseFormatter:
    def format_response_with_context(self, llm_response, retrieval_context):
        """Format response with natural document attribution"""
        
        # Extract document context
        doc_context = retrieval_context['document_context']
        chunks_used = retrieval_context['chunks']
        
        # Group chunks by document for attribution
        chunks_by_doc = self.group_chunks_by_document(chunks_used)
        
        # Build attribution section
        attribution = self.build_natural_attribution(chunks_by_doc, doc_context)
        
        # Format final response
        formatted_response = f"""{llm_response}

{attribution}"""
        
        return formatted_response
    
    def build_natural_attribution(self, chunks_by_doc, doc_context):
        """Build natural language source attribution"""
        if len(chunks_by_doc) == 1:
            doc_id = list(chunks_by_doc.keys())[0]
            doc_info = doc_context[doc_id]
            return f"\n📄 **Based on {doc_info['filename']}**: {doc_info['summary']}"
        
        elif len(chunks_by_doc) <= 3:
            attributions = []
            for doc_id, chunks in chunks_by_doc.items():
                doc_info = doc_context[doc_id]
                section_info = self.identify_key_sections(chunks)
                attributions.append(f"📄 **{doc_info['filename']}** ({section_info})")
            
            return f"\n**Sources:**\n" + "\n".join(attributions)
        
        else:
            return f"\n**Sources**: Information compiled from {len(chunks_by_doc)} project documents"
    
    def identify_key_sections(self, chunks):
        """Identify which sections of document were used"""
        if all(chunk.get('section_type') for chunk in chunks):
            sections = list(set(chunk['section_type'] for chunk in chunks))
            return f"Sections: {', '.join(sections[:3])}"
        else:
            return f"{len(chunks)} relevant sections"

# Chapter 8: User Authentication & Team Features
Core Philosophy: Glass Office Collaboration Through Natural Boundaries
Giljo's user authentication and team features embody the glass office philosophy - transparency and collaboration within natural organizational boundaries. Rather than imposing complex enterprise permission matrices, GIljo recognizes how SMB teams actually work: departments create logical collaboration spaces, projects focus work initiatives, and people naturally understand who should see what.
The system eliminates the privacy-vs-collaboration trade-off that plagues enterprise solutions. Users see everything relevant to their work while maintaining clear, simple boundaries that protect sensitive information when needed.
Strategic Authentication Architecture
Four-Tier Cascading Role System
SMB organizations require flexibility that rigid enterprise roles cannot provide. GIljo implements a cascading hierarchy where each tier inherits all capabilities of lower tiers:
Owner: Full System Access

Complete administrative control
Can perform all Admin, Super User, and User functions
Data export, system reset, and advanced configuration
Ultimate decision authority on organizational policies

Admin: Day-to-Day Management

User management and department assignment
System configuration and maintenance oversight
Can perform all Super User and User functions
Handles routine administrative tasks without Owner intervention

Super User: Cross-Department Project Creation

Can create projects in any department (addresses SMB multi-hat reality)
Full User capabilities plus expanded project creation rights
Maintains "home department" for organizational purposes
Perfect for roles like HR/Marketing or Owner/Sales combinations

User: Department-Scoped Collaboration

Can create projects within assigned department
Participates in projects via invitation system
Core collaboration and document interaction capabilities
Foundation tier that all others build upon

The Multi-Hat Problem Solved
In SMB environments, the HR person often handles marketing, and the owner wears multiple functional hats. Super Users can create projects in any department, reflecting SMB work reality rather than forcing artificial organizational constraints.
Implementation Logic: This cascading system eliminates permission complexity while enabling natural work patterns. A Super User assigned to Marketing can create projects in Legal when wearing their compliance hat, but still maintains Marketing as their "home department" for organizational purposes.

# Chapter 9: Privacy & Project Management - Implementation Guide

Context & Design Philosophy
? Deferred Chapter: Advanced privacy features for organizations requiring confidential project management
? Target: Professional services with sensitive client work, executive planning, intellectual property
? Approach: Simple binary privacy model (public vs private projects) without complex enterprise permissions
? Foundation: Builds on Chapter 8's glass office collaboration with privacy layer overlay
Core Privacy Model - Simple Binary Approach
1. Project Privacy Types
Public Projects (Default):
- Full glass office collaboration within departments
- Visible in navigation tree to all department members
- Files and chats searchable via department search toggle
- Standard team invitation and collaboration features

Private Projects:
- Invitation-only access with explicit member management
- Completely isolated from department search
- Files remain private to project members only
- Visible in navigation tree but access restricted

2. Project Creation Privacy Decision
User Interface:
```
Create New Project
Project Name: [Contract Review 2024]
Department: [Legal]

? Make this a private project
(Only invited members can access files and conversations)
```

Default: Unchecked (public project encouraging collaboration)
Privacy by Choice: Users consciously decide when privacy is needed

3. File Management Privacy Rules
Public Files in Public Projects:
- Appear in Main File Manager
- Discoverable via Universal Search
- Available for cross-project assignment
- Standard glass office behavior

Private Files in Private Projects:
- Do NOT appear in Main File Manager
- Only visible within the private project
- Not discoverable via Universal Search
- Completely isolated from public space

Public Files Added to Private Projects:
- Remain public and visible in Main File Manager
- Project shows ?Unlocked [icon] Public File" indicator
- Maintains cross-project availability
- No privacy restrictions applied

Advanced Privacy Features

4. File Privacy Management
Within Private Project File Manager:
Make File Public Button:
- Moves private file to Main File Manager
- Makes discoverable via Universal Search
- Irreversible privacy change (requires confirmation)
Assign to Another Project:
- Share private files between private projects
- User dropdown ? Project dropdown selection
- File remains private but accessible to multiple project teams

5. Project Deletion Privacy Handling

```
This project contains 5 private files:

What should happen to these files?

[Delete Project] [Cancel]
```

6. Navigation and Discovery

- Indicates existence without granting access

- Private projects: "Make project description searchable" checkbox (default OFF)

- Conservative privacy by default

### Access Control Logic
**File Visibility Rules:**
```python
def file_visible_to_user(file, user):
    if file.is_public:
        return True
    
    # Private file - check project membership
    for project in file.projects:
        if project.is_private and user in project.members:
            return True
    
    return False
```
