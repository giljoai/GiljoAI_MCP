+# Manual Test Runbook: Claude.ai Handshake
+
+**Who runs this:** The reviewer (or Patrik) before merging any PR labeled `manual-test:claude-ai-handshake`.
+**Time required:** ~10 minutes with a running demo instance.
+
+---
+
+## 1. Purpose
+
+This runbook covers the end-to-end OAuth handshake between Claude.ai and the GiljoAI MCP demo server. It catches regressions that automated tests cannot reach because they live in the intersection of:
+
+- The browser `authGuard.js` redirect logic (frontend routing)
+- The OAuth authorize endpoint (`/oauth/authorize`) parameter-forwarding
+- The consent-screen rendering (`OAuthAuthorize.vue`, `DemoConsentScreen.vue`)
+- The Claude.ai-side OAuth client library (external, no API access)
+
+**Motivation:** On 2026-05-09, a frontend authGuard regression silently dropped `state` and `code` query parameters on the redirect back to Claude.ai. Claude.ai received a parameter-stripped URL and showed "Failed to connect." — a failure invisible to backend tests, unit tests, and linters. API-0021d (2026-05-10) fixed the root cause; this runbook ensures the same class of bug is caught by a human reviewer before it ships.
+
+---
+
+## 2. Pre-requisites
+
+- Access to **`demo.giljo.ai`** (the live demo) OR a local dev instance with `GILJO_MODE=demo` and `demo/seed_demo_data.py` already run (seeds the `claude.ai-demo-shared` OAuth client).
+- A Claude.ai account with the **MCP server connection** feature enabled. (As of 2026-05, this is in Claude.ai Pro / Team under Settings → Integrations → MCP Servers. If you don't see it, the feature is not yet enabled on your account — test on a different account.)
+- A browser where you can open Claude.ai and inspect the URL bar at each redirect step.
+
+---
+
+## 3. Test Steps
+
+1. **Open Claude.ai** → Settings → Integrations → MCP Servers → Add server.
+
+2. **Enter the server URL.** For demo: `https://demo.giljo.ai/mcp`. For local dev: `https://127.0.0.1:7272/mcp` (or whichever port your LAN instance is on). Click Connect / Save.
+
+3. **Observe the redirect.** Claude.ai should redirect your browser to the GiljoAI authorize page. The URL in your browser bar should look like:
+   ```
+   https://demo.giljo.ai/oauth/authorize?response_type=code&client_id=claude.ai-demo-shared&redirect_uri=...&scope=...&state=<random-string>
+   ```
+   Confirm: `state=` is present and non-empty. Confirm: `client_id=claude.ai-demo-shared`.
+
+4. **Consent screen renders.** The page should display:
+   - Full application name (e.g. "Claude.ai")
+   - The list of requested scopes (e.g. "Read projects", "Create tasks", "Agent access")
+   - A "Connect" (or "Authorize") button
+   - No blank page, no 401, no "missing parameters" error.
+
+5. **Click Connect.** The page processes and redirects back to Claude.ai. Watch the URL bar during the redirect — the outgoing URL to Claude.ai should contain **both** `state=<same-string-as-step-3>` **and** `code=<auth-code>`.
+
+6. **Claude.ai shows "Connected."** After the redirect completes, Claude.ai settings should update to show the server as connected with a green indicator. If it shows "Failed to connect" or prompts to retry, the handshake failed — see Section 5.
+
+7. **Open a new chat.** Start a fresh Claude.ai conversation. Type a prompt that calls an MCP tool, for example: `List my GiljoAI projects`.
+
+8. **Verify MCP tool response.** Claude.ai should invoke the MCP tool and return a valid response (project list, or "no projects found" — either is fine). A 401, 403, or tool-not-found error is a failure.
+
+9. **Reconnect test (rebase resilience).** If the PR introduced changes to the consent screen or redirect logic, disconnect the server in Claude.ai settings and repeat steps 2-8 once more to confirm idempotency.
+
+---
+
+## 4. Pass Criteria
+
+| Step | Observable signal |
+|------|-------------------|
+| Step 3 | URL bar shows `/oauth/authorize?...&state=<value>` — `state` is non-empty |
+| Step 4 | Consent screen shows client name + scope list; no error banner |
+| Step 5 | Outgoing redirect URL contains both `state=<same-value>` and `code=<value>` |
+| Step 6 | Claude.ai shows "Connected" with green indicator |
+| Step 8 | MCP tool call returns a valid response (not 4xx/5xx) |
+
+All five must pass. One failure = do not merge.
+
+---
+
+## 5. Fail Criteria (Common Failure Signatures)
+
+| Symptom | Likely cause |
+|---------|--------------|
+| "missing parameters" error after clicking Connect | `authGuard.js` dropped `state` or `code` from the redirect URL (same class as 2026-05-09 bug) |
+| Redirect loop on `/oauth/authorize` | `redirect_uri` mismatch or CSRF token failure |
+| Blank consent screen | `OAuthAuthorize.vue` fetch failed; check browser console for 401/500 |
+| 401 on `/mcp` endpoint | OAuth token not stored; check cookie / session after step 6 |
+| Claude.ai shows "Failed to connect" | Any of the above; check network tab for the final redirect URL — confirm `state` and `code` are present |
+
+---
+
+## 6. What to Do on Failure
+
+1. **Do not merge the PR.**
+2. Open a **P0 issue** in the private repo with:
+   - The exact URL observed at the failing step (copy from browser bar)
+   - A screenshot of the error state
+   - The PR number that triggered the failure
+3. Tag the PR author and link the issue in a PR comment.
+4. The issue must be resolved and this runbook re-run successfully before the PR can be merged.
+

+## 8. Branch Protection Note
+
+Patrik must configure a branch protection rule in GitHub Settings to enforce the `manual-test:verified` label before merge. This is a UI-only step — it cannot be done in YAML.
+
+**Path:** GitHub → Repository → Settings → Rules → Rulesets → master-protection (or the equivalent private-repo ruleset) → Add rule → "Require label" → `manual-test:verified`.
+
+Until this rule is enabled, the label is advisory only — reviewers must manually check for it before approving the merge.
