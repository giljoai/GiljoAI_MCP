Vision Document Analysis GiljoAI MCP — Lean Implementation Spec v2.0

MCP-native field extraction + Sumy fallback. No provider architecture. No proposal workflow.
Target: CE launch, end of April 2026. Estimated effort: ~8 agent-hours.

Design principle: GiljoAI is a passive orchestrator. It stores the document, provides the extraction prompt, and exposes an MCP write tool. The user's AI coding tool does the reasoning. GiljoAI never runs inference.

3. MCP Tools — Two Tools Total

3.1 /gil_get_vision_doc

Returns the vision document content with the baked extraction prompt. The agent's LLM uses this to understand what fields to extract.

# MCP Tool Definition
{
  "name": "gil_get_vision_doc",
  "description": "Retrieve a product's vision document with extraction instructions.
    Use this when asked to analyze a vision document and populate product fields.
    The response includes the document content and a structured extraction prompt
    that tells you exactly what fields to extract and how to format them.",
  "parameters": {
    "product_id": {
      "type": "string",
      "required": true,
      "description": "The product ID to retrieve the vision document for."
    }
  }
}

# Server-side: returns
{
  "document_content": "... full document text or chunked ...",
  "document_tokens": 18500,
  "extraction_instructions": "... the baked prompt (see Section 4) ...",
  "write_tool": "gil_write_product",
  "product_id": "abc-123",
  "product_name": "TinyContacts"
}
Key design: The extraction prompt is baked into the tool response, not into the agent's system prompt. This means GiljoAI controls the prompt. The user's agent just follows the instructions it receives. If you improve the prompt later, every user gets the upgrade automatically.

3.2 /gil_write_product

One flat tool. Every product field is an optional parameter. Only fields present in the call get written. Includes summary fields.

4. The Baked Extraction Prompt

This prompt ships with GiljoAI. It is returned by /gil_get_vision_doc as part of the response, not stored in the agent's config. Users can tune it via a "Custom Instructions" text field in product settings (appended, not replaced). 
# src/giljo_mcp/prompts/vision_extraction.py

VISION_EXTRACTION_PROMPT = """You are analyzing a product vision document for a software
development orchestration platform. Extract structured information and generate summaries.

RULES:
- Extract ONLY information explicitly stated in the document
- If a field cannot be determined from the document, OMIT it entirely
- Do NOT guess, invent, or infer information not present
- Keep descriptions concise and factual, not promotional
- product_description should be 2-3 sentences maximum
- For summaries: preserve technical specs, architecture decisions, and constraints
- For summaries: remove marketing prose, user personas, and storytelling

After reading the document, call the gil_write_product tool with all fields you were
able to extract. Include summary_33 (concise ~33% executive summary focusing on what
a developer needs to build this) and summary_66 (thorough ~66% technical summary
preserving decisions, architecture, and feature descriptions).

{custom_instructions}

Here is the document to analyze:

{document_content}"""

The prompt instructs the agent to call gil_write_product directly. The agent's LLM reads the document, reasons about it, and makes the tool call. GiljoAI never touches the document content with its own inference — the user's AI tool does all the reasoning.

5. Database Changes

5.1 Summaries Table (new)

CREATE TABLE vision_document_summaries (
    id VARCHAR(36) PRIMARY KEY,
    tenant_key VARCHAR(255) NOT NULL,
    document_id VARCHAR(36) NOT NULL,       -- FK to vision_documents
    product_id VARCHAR(36) NOT NULL,        -- FK to products
    source VARCHAR(20) NOT NULL,            -- "sumy" | "ai"
    ratio DECIMAL(3,2) NOT NULL,            -- 0.33 | 0.66
    summary TEXT NOT NULL,
    tokens_original INTEGER NOT NULL,
    tokens_summary INTEGER NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),

    INDEX idx_vds_lookup (tenant_key, document_id, source, ratio),
    FOREIGN KEY (document_id) REFERENCES vision_documents(id) ON DELETE CASCADE,
    FOREIGN KEY (product_id) REFERENCES products(id) ON DELETE CASCADE
);

5.2 Product Settings Addition

-- Add to existing product settings or product table:
-- extraction_custom_instructions TEXT  (nullable, user's custom prompt additions)

-- No new tables for proposals. No status tracking.
-- Fields are just product fields. Summaries are just summaries.

5.3 Context Manager Integration

-- Context Manager reads summaries for depth_config:
--
-- depth_config: { "vision_documents": "full" }
--   → serve original document from vision_documents table
--
-- depth_config: { "vision_documents": "medium" }
--   → SELECT summary FROM vision_document_summaries
--     WHERE ratio = 0.66
--     ORDER BY source = 'ai' DESC, created_at DESC  -- prefer AI over Sumy
--     LIMIT 1
--
-- depth_config: { "vision_documents": "light" }
--   → same query but ratio = 0.33
--
-- If no AI summary exists, Sumy summary is served (always available).

10. Edge Cases

No CLI tool connected yet

"Stage Analysis" button shows but with helper text: "Requires a connected AI coding tool. Set up MCP connection →". Sumy summaries are always available regardless. User can fill fields manually and connect their tool later.

Document exceeds 20K tokens

Existing chunker splits the document. /gil_get_vision_doc returns all chunks. The agent's LLM processes them. If the agent's context window can't handle all chunks, the agent is responsible for its own chunking strategy — GiljoAI provides the content, the agent decides how to consume it. This is consistent with how every other MCP context delivery works.

Agent writes bad data

The user reviews and edits in the web UI. Same as if they typed bad data themselves. No special handling needed. The instruction in the prompt ("OMIT fields you can't determine, do NOT guess") reduces this risk, but ultimately HITL is the safety net and it's the existing edit form.

User re-uploads a new version

New document replaces old. Sumy reruns. If user clicks "Stage Analysis" again, agent overwrites fields with new extraction. Previous summaries are replaced. This is the expected behavior — the user uploaded a new doc, they want fresh analysis.
