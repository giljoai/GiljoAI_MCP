## Task Summary

Two protocol enhancements that get injected into agent missions/protocols for multi-terminal mode:

**A) /gil_add guidance at completion**: Multi-terminal agents each have their own user-facing terminal. When an agent completes, it should tell the user about `/gil_add` for saving follow-up projects or technical debt. This does NOT apply to CLI sub-agents (they don't have user-facing terminals).

**B) Git commit instruction**: When the user has git integration enabled (first toggle in user settings), agent protocols should instruct agents to commit their work before completing. This provides a traceable artifact for successor agents and the orchestrator. When git is NOT enabled, agents rely only on the completion result dict (0497b) for traceability.

## Context and Background

### /Gil_add Context
`/gil_add` is a user-invokable skill (slash command) that routes to either task (technical debt/TODOs) or project (actionable work items) on the GiljoAI dashboard. In multi-terminal mode, each agent terminal is a user-facing session. When an agent finishes, it should offer:

> "My work is complete. If you'd like to save any follow-up work or technical debt I discovered, tell me and I'll use /gil_add to save it to your dashboard."

In CLI mode, sub-agents are Task()-spawned and have no user terminal — they report to the orchestrator, which handles /gil_add. So this injection is **multi-terminal only**.

### Git Integration Context
The `git_integration` toggle exists in **two separate locations** in the codebase:

1. **Product-level (per-product):** `product.product_memory["git_integration"]["enabled"]` — JSONB field on the Product model. Set via `ProductService.update_git_integration()`. Used by `thin_prompt_generator._inject_git_instructions()` and the `get_git_history` context tool.

2. **Global config:** `config.yaml` → `features.git_integration.enabled`. Read via `get_config().get_nested("features.git_integration.enabled", default=False)`. Currently used by the orchestrator mission builder at `orchestration_service.py:3104`.

**Recommended approach:** Use `get_config().get_nested("features.git_integration.enabled")` for simplicity — it avoids needing to load the Product model in `get_agent_mission()`. This matches the pattern already used by the orchestrator mission builder.

When git integration is enabled, agents should be instructed to commit their work. The commit message and hash become part of the completion result dict (0497b), providing a traceable artifact.

### Where Protocol Gets Injected
The agent lifecycle protocol is generated by `_generate_agent_protocol()` — a **module-level function** (not a method on OrchestrationService) in `orchestration_service.py` at line 209. It has NO `self` and NO database session access. Current signature: `(job_id: str, tenant_key: str, agent_name: str, agent_id: str | None = None) -> str`. It is called by `get_agent_mission()` (line ~1806) and returns the `full_protocol` field. The protocol has 5 phases — we're modifying Phase 4 (COMPLETION).

**Important:** Since this is a module-level function with no DB access, `execution_mode` and `git_integration` must be resolved by the caller (`get_agent_mission()`) and passed as new parameters. The project is already loaded in `get_agent_mission()` at line ~1655 so `project.execution_mode` is available. For git_integration, use `get_config().get_nested("features.git_integration.enabled")`.

**Note:** `spawn_agent_job()` (line 1198) branches on `execution_mode` for template injection at line ~1378. The git commit instruction should be injected in `_generate_agent_protocol()` (generated dynamically at mission read time), NOT at spawn time (persisted in DB). The protocol approach is cleaner because it can be toggled without re-spawning agents.

### Files to Modify

**`src/giljo_mcp/services/orchestration_service.py`** — `_generate_agent_protocol()`:

Modify Phase 4 (COMPLETION) section. The actual Phase 4 is more detailed than shown below (includes full MCP call signatures with f-string substitutions for `{executor_id}`, `{tenant_key}`, `{job_id}`, plus rejection warning). Abbreviated for readability:
```
### Phase 4: COMPLETION
Before calling `complete_job()`, you MUST verify:
1. All TODO items completed (your TodoWrite list is fully marked completed)
2. All messages read (queue empty after `receive_messages()`)

Final steps:
1. Call `receive_messages()` - Final message check (full MCP call with agent_id, tenant_key)
2. Process any pending messages - ensure queue is empty
3. Call `complete_job()` - ONLY after TODOs complete and queue empty (full MCP call with job_id, result dict)
```

**F-string escaping note:** Phase 4 uses heavy f-string interpolation. Literal braces in injected text (e.g., `result={"summary": ...}`) already use `{{` / `}}` escaping. The implementer must maintain this pattern in any new conditional blocks.

Add two conditional blocks:

**Git commit block** (injected when `git_integration` is enabled):
```
### Git Commit (REQUIRED - Git Integration Enabled)
Before calling `complete_job()`, commit your work:
1. Stage your changes: `git add` relevant files (never use `git add -A`)
2. Write a descriptive commit message summarizing your work
3. Include the commit hash in your completion result:
   `complete_job(job_id, result={"summary": "...", "commits": ["abc123"], "artifacts": [...]})`
```

**/gil_add block** (injected when `execution_mode == "multi_terminal"`):
```
### User Guidance (Multi-Terminal)
After completing your work, tell the user:
"My work is complete. If you discovered technical debt or want to save follow-up work,
tell me and I'll use /gil_add to save it to your dashboard."
```
