## Executive Summary

**Problem**

Alpha trial revealed critical message handling gaps in the agent lifecycle protocol:
1. **Agents not reading all messages** - Some agents were missing orchestrator guidance messages sent during their execution
2. **Orchestrator not checking messages during implementation** - Orchestrator would spawn agents and wait for completion without polling for status updates or error reports

**Root Cause**

The `full_protocol` generated in `orchestration_service.py` (lines 153-211) provides a 5-phase lifecycle but lacks explicit, emphatic guidance on:
- When to check for messages during execution phases
- The critical difference between `receive_messages()` (auto-acknowledging) vs `list_messages()` (read-only)
- Mandatory message checks before phase transitions
- How orchestrators should monitor agent progress during parallel/sequential execution

**Impact**

Without comprehensive message handling:
- Agents may miss critical course corrections from orchestrator
- Orchestrators cannot detect blocked agents in real-time
- Team coordination breaks down (agents working in isolation)
- Error states may go undetected until completion timeout

**Solution**

Enhance protocol generation to include explicit message handling instructions for:
1. **Generic agent protocol** - Clear "when" and "how" for message checks
2. **Orchestrator execution prompts** - Message polling patterns for sequential/parallel spawning
3. **Phase transition gates** - Mandatory message checks before advancing

## Context & Background

### Alpha Trial Findings

During the TinyContacts project alpha trial (Issue 0361), we observed:

1. **Issue #2: Agents Not Reading Messages**
   - Orchestrator sent guidance messages to spawned agents
   - Some agents never called `receive_messages()` to check for them
   - Messages remained in pending state until manual intervention
   - Root cause: Protocol suggests message checking but doesn't mandate it

2. **Issue #8: Orchestrator Not Monitoring During Implementation**
   - Orchestrator spawned analyzer and documenter agents
   - Went silent during agent execution (no progress polling)
   - Did not detect when documenter completed before analyzer
   - Only checked results at final completion phase
   - Root cause: Staging prompt lacks execution-phase monitoring guidance

### Current Protocol (Lines 172-211)

The current `_generate_agent_protocol()` includes:
- Phase 1 (STARTUP): Call `receive_messages()` once after acknowledging job
- Phase 3 (PROGRESS REPORTING): Call `receive_messages()` after each milestone
- Note at lines 195-198: "MESSAGE HANDLING (CRITICAL - Issue 0361-5)" explaining the difference between `receive_messages()` and `list_messages()`

**What's Missing:**
- No explicit instruction to check messages DURING Phase 2 (EXECUTION)
- No guidance on message check frequency (continuous vs milestone-based)
- No mandatory "empty queue before continuing" gates
- No orchestrator-specific patterns for monitoring spawned agents

### Architecture Context

**Message Queue System** (Handover 0362):
- Messages stored in `MCPAgentJob.message_queue` JSONB column
- `send_message()` adds messages to recipient's queue
- `receive_messages()` auto-acknowledges and removes from queue
- `list_messages()` reads without acknowledgment (debugging only)

**Agent Lifecycle** (Handover 0334, 0359):
- 5-phase protocol embedded in `get_agent_mission()` response
- Phase 1: STARTUP (acknowledge, check messages, plan tasks)
- Phase 2: EXECUTION (perform work)
- Phase 3: PROGRESS REPORTING (update progress, check messages)
- Phase 4: COMPLETION (finalize and report results)
- Phase 5: ERROR HANDLING (report blocks and await guidance)

**Orchestrator Workflow** (Handover 0246a):
- 7-task staging workflow (identity, health check, agent discovery, spawning, activation)
- Staging prompt generated by `ThinClientPromptGenerator._build_staging_prompt()`
- Execution phase NOT explicitly defined in current staging prompt

## Problem Statement

### Problem 1: Agent Message Blind Spots

**Current Behavior:**
- Agents check messages at startup (Phase 1) and milestones (Phase 3)
- Long execution blocks (Phase 2) may span 15-30 minutes without message checks
- Orchestrator guidance sent mid-execution may go unread until next milestone

**Example Scenario:**
```
T+0min:  Analyzer starts, checks messages (empty queue)
T+2min:  Orchestrator sends "Focus on data model first"
T+5min:  Analyzer still working (hasn't checked messages)
T+15min: Analyzer completes Phase 2, reports progress (finally reads message)
Result:  Wasted 13 minutes on wrong priority
```

**Expected Behavior:**
- Agents check messages at regular intervals during execution
- Empty queue = safe to continue working
- Non-empty queue = process feedback BEFORE continuing

### Problem 2: Orchestrator Execution Blindness

**Current Behavior:**
- Orchestrator spawns agents and waits passively
- No guidance on when to poll agent status during execution
- No pattern for handling "agent finished but another still working" scenarios

**Example Scenario:**
```
T+0min:  Orchestrator spawns Analyzer + Documenter (parallel)
T+5min:  Documenter finishes, waits idle
T+15min: Analyzer still working
T+15min: Orchestrator still in passive wait state
T+20min: Analyzer finishes
T+20min: Orchestrator finally checks results
Result:  No opportunity to send mid-flight corrections or coordinate handoffs
```

**Expected Behavior:**
- **Sequential mode**: Check messages after each agent completes
- **Parallel mode**: Poll for completion + check messages as agents finish
- **Before final completion**: Mandatory message check for any blocking issues

## Risks & Mitigations

### Risk 1: Over-Polling (Performance Degradation)

**Risk**: Agents checking messages too frequently may slow execution
**Likelihood**: Medium
**Impact**: Low (message check is <100ms operation)
**Mitigation**:
- Recommend "after each task" frequency (not time-based polling)
- For long tasks (>10 min), suggest explicit "check every 5 min" in mission text
- Monitor message check frequency in production logs

### Risk 2: Token Budget Overflow

**Risk**: Additional protocol text pushes agents over context budget
**Likelihood**: Low
**Impact**: Medium (would require protocol compression)
**Mitigation**:
- Current protocol: ~1,400 tokens
- Proposed additions: +200 tokens
- New total: ~1,600 tokens (well under 2K safe limit)
- If needed, can compress other sections (e.g., MCP tool examples)

### Risk 3: Agent Confusion (Too Many Instructions)

**Risk**: Agents misinterpret "check after each task" and over-check
**Likelihood**: Low
**Impact**: Low (extra checks are harmless)
**Mitigation**:
- Use clear, unambiguous language ("after completing each TodoWrite task")
- Provide examples in protocol text
- Monitor agent behavior in alpha trials for confusion patterns

### Risk 4: Breaking Changes to Existing Jobs

**Risk**: Protocol changes break in-flight agent jobs
**Likelihood**: Low
**Impact**: Medium
**Mitigation**:
- Protocol is generated per-job (no retroactive changes)
- Existing jobs use old protocol until completion
- New jobs use enhanced protocol from creation
- No database migration required

## Notes & Observations

1. **Design Decision: "After Each Task" vs Time-Based Polling**
   - Chose task-based frequency over time-based (e.g., "every 2 minutes")
   - Rationale: Task boundaries are natural checkpoints, time-based adds complexity
   - Tasks typically complete in 1-5 minutes, providing adequate responsiveness

2. **Design Decision: Mandatory Gates vs Suggestions**
   - Used strong language ("MANDATORY", "MUST", "Do NOT proceed if...")
   - Rationale: Agents interpret suggestions loosely; directives are followed
   - Alternative considered: Implement server-side enforcement (rejected as over-engineering)

3. **Token Budget Considerations**
   - Agent protocol grew from ~1,400 to ~1,600 tokens (+14%)
   - Orchestrator staging grew from ~930 to ~1,180 tokens (+27%)
   - Both remain well under 2K token soft limit
   - If budget becomes issue, can compress MCP tool examples or move to external docs

4. **Future Enhancement Opportunities**
   - Add server-side staleness detection (flag messages unread >10 minutes)
   - Implement escalation (send WebSocket alert if agent ignores messages)
   - Add orchestrator dashboard showing real-time message queue status
   - Consider rate limiting (prevent message spam from overwhelming agents)
