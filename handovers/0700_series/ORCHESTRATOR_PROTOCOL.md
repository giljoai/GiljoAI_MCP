# 0700 Series Orchestrator Protocol

**Purpose**: Coordinate sequential code cleanup handovers across multiple terminal sessions with full context continuity.

---

## How This Works

### Your Role (Orchestrator Session)

You keep ONE terminal session open as the "orchestrator". This session:
1. Reads the master state (`orchestrator_state.json`)
2. Identifies the next handover to execute
3. Generates a kickoff prompt for a NEW terminal session
4. Waits while you run the worker session
5. Reads results (comms_log, state updates)
6. Generates the next kickoff prompt
7. Repeats until series complete

### Worker Sessions (You Launch Separately)

For each handover, you:
1. Open a NEW Claude Code terminal
2. Paste the kickoff prompt generated by orchestrator
3. Let the worker execute with subagents
4. Worker writes to comms_log and updates state
5. Worker commits changes
6. Close worker terminal
7. Return to orchestrator terminal

## Orchestrator Commands

### Start Orchestrator Session

In your orchestrator terminal, say:

```
I'm starting the 0700 code cleanup orchestrator session. Read the orchestrator state
at handovers/0700_series/orchestrator_state.json and generate the kickoff prompt
for the first pending handover. Save the kickoff prompt to the kickoff_prompts folder.
```

### After Each Worker Completes

Return to orchestrator terminal and say:

```
Worker session for handover [0700/0701/etc] is complete. Read the comms_log and
orchestrator_state to see what happened. Generate the kickoff prompt for the
next pending handover.
```

### Check Series Progress

```
Show me the current status of the 0700 code cleanup series. Read orchestrator_state.json
and comms_log.json and give me a summary.
```

### Handle Blockers

```
There's a blocker from the worker session. Read the comms_log for entries with
type='blocker' and help me decide how to proceed.
```

## Worker Session Protocol

Each worker session follows this protocol:

### Phase 1: Context Acquisition
1. Read orchestrator_state.json for series context
2. Read comms_log.json for notes from previous handovers
3. Read doc_impacts.json to know which docs might need updates
4. Read the specific handover spec (e.g., handovers/0700_code_cleanup_index.md)

### Phase 2: Scope Investigation
1. Scan codebase for items in scope (DEPRECATED markers, etc.)
2. Identify all files that will be modified
3. Map files to documentation that references them
4. Create execution plan

### Phase 3: Execution
1. Execute cleanup with appropriate subagents (database-expert, tdd-implementor, etc.)
2. Run tests after each significant change
3. Validate no cascading breaks
4. Track all changes made

### Phase 4: Documentation
1. Review doc_impacts.json for docs that need updates
2. Update documentation to reflect code changes
3. Record updates in doc_impacts.json

### Phase 5: Communication
1. Write entries to comms_log.json for subsequent handovers
2. Include warnings, suggestions, dependencies discovered
3. Update orchestrator_state.json (mark complete, record docs updated)

### Phase 6: Commit
1. Stage all changes (code + docs)
2. Create atomic commit with clear message
3. Report completion summary

## Comms Log Entry Types

| Type | When to Use |
|------|-------------|
| `info` | General information for future agents |
| `warning` | Something to be aware of but not blocking |
| `blocker` | Critical issue that MUST be addressed before continuing |
| `dependency` | A change that creates dependency for later handover |
| `suggestion` | Recommendation for how to handle something |

## Kickoff Prompt Template

When generating kickoff prompts, the orchestrator uses this structure:

```markdown
# Handover [ID]: [Title]

## Series Context
You are executing handover [ID] in the 0700 Code Cleanup Series.
- Series progress: [X/11 complete]
- Previous handover: [ID] - [Title]
- Next handover: [ID] - [Title]

## Your Mission
[Specific scope and tasks from handover spec]

## Context from Previous Agents
[Relevant entries from comms_log.json]

## Documentation to Review/Update
[Relevant entries from doc_impacts.json]

## Execution Protocol
1. Read the full handover spec at: handovers/[handover_file].md
2. Follow the Worker Session Protocol in ORCHESTRATOR_PROTOCOL.md
3. Use subagents: [recommended agents for this handover]
4. When complete, update:
   - comms_log.json (notes for future agents)
   - orchestrator_state.json (mark complete)
   - doc_impacts.json (record doc updates)

## Communication Requirements
Before completing, write to comms_log.json with:
- Any warnings for handovers [subsequent IDs]
- Files you modified that they might touch
- Patterns you established they should follow

## Completion Checklist
- [ ] All cleanup tasks completed
- [ ] Tests passing
- [ ] Documentation updated
- [ ] Comms log entries written
- [ ] State file updated
- [ ] Changes committed
```

## Getting Started

1. **Open orchestrator terminal**
2. **Say**: "I'm starting the 0700 code cleanup orchestrator. Read the state and generate the first kickoff prompt."
3. **Copy** the generated kickoff prompt
4. **Open new terminal** for worker session
5. **Paste** kickoff prompt
6. **Let worker complete**
7. **Close worker terminal**
8. **Return to orchestrator terminal**
9. **Say**: "Worker for 0700 complete. Generate next kickoff."
10. **Repeat** until series complete

## Recovery Procedures

### Worker Session Fails Mid-Execution

1. Note where it failed
2. Return to orchestrator
3. Say: "Worker for [ID] failed during [phase]. Read comms_log for any partial updates. Help me recover."

### Need to Skip a Handover

1. Return to orchestrator
2. Say: "I need to skip handover [ID] because [reason]. Update the state and generate kickoff for next handover."

### Need to Re-run a Handover

1. Return to orchestrator
2. Say: "Handover [ID] needs to be re-run because [reason]. Reset its state and generate a new kickoff prompt."
